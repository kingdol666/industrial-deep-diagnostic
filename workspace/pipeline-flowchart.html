📋 8步流水线 · 每步详解

Step 0 · Setup Workspace Main Agent

创建运行目录结构 + Python环境
  • 执行 setup.mjs 创建 <timestamp>_<scene>/ 目录
  • 自动生成7个子目录: 00_input ~ 06_scripts
  • 执行 uv_env_setup.mjs 创建 Python venv
  • 安装依赖: matplotlib, numpy, pandas, seaborn, scipy, openpyxl
  • ⚠️ 所有Python脚本 必须 使用 venv python，禁止系统python3

Step 1 · Inspect Data Main Agent

数据检查 + 用户上下文采集
  • 执行 inspect.mjs 扫描数据: 列名、类型、范围、缺失值
  • 向用户提问 ≤5个澄清问题: 工艺类型、质量问题、已知参数
  • 输出 input_manifest.json 数据清单
  • 输出 user_context.json 用户上下文
  • 场景名称 = 工艺描述，批次信息等

Step 2 · Context Builder Sub-Agent

领域知识提取 + 工业本体构建
  • 搜索 REFERENCE_DIR 中的SOP/手册/报告
  • 可选: 最多5次Web搜索补充领域知识
  • 构建 ontology.json: 场景、设备、工艺阶段、参数物理含义
  • 构建 schema.json: 参数分类 (control/process/quality..)
  • 输出 clarification_needed.json: 未知物理含义的参数
  • 输出 extracted_knowledge.json: 已知故障模式、因果关系

Step 2.5 · Clarification Gate Main Agent

参数含义澄清门
  • 检查 clarification_needed.json 中的未知参数
  • 若存在 CRITICAL/HIGH 级别未知 → 向用户提问
  • 参数含义确认后更新 ontology.json
  • 若无未知参数 → 直接进入 Step 3
  • ⚠️ 未确认物理含义的参数,诊断置信度上限=50

Step 3 · Data Processor Sub-Agent

数据处理 + 统计分析 + 可视化 + 验证
  • 场景分类: CNC / 连续 / 批次 / 热交换 / 质量 / 通用
  • 预处理: 缺失值、异常值(IQR)、排序验证、衍生特征
  • 运行 stats.mjs → feature_summary.json
  • 运行 stats_validate.mjs → validate_report.json
  • 异常检测 → anomaly_report.json
  • 因果证据图 → causal_evidence_map.json
  • 生成 场景自适应 可视化 → 03_figures/*.png
  • 所有JSON输出需要 Schema 验证

Step 4 · Diagnostician Sub-Agent

竞争假设诊断 · 根因推理
  • Phase 1 - 数据探针: 直接探测原始数据拐点、过渡事件
  • Phase 2 - 分层分析: 验证子组内相关性是否成立
  • 5步协议:
    A. 假设生成 (物理链+可视化证据)
    B. 交叉验证 (数据探针+因果图)
    C. 可区分性评估 (DISCRIMINABLE/INDISTINGUISHABLE)
    D. 排除 (物理不可能+数据矛盾)
    E. 结论: DETERMINED / COMPETING_SET / NEEDS_DATA
  • 输出: reasoning_chain.json (R1-R8) + diagnosis.json + evidence.json + confidence.json

Step 5 · Judge Review Sub-Agent

最终质量门 · 10维度评分
  • 交叉验证 validate_report.json 与 diagnosis.json
  • 检查: 排序问题、Simpson悖论、趋势混杂、异常值驱动
  • 检查: 参数物理含义标记 ([PARAM_AMBIGUITY])
  • 检查: 可区分性评估 → INDISTINGUISHABLE置信度≤65
  • 评分结果:
    ✅ PASS ≥90 → 进入 Step 6
    🔄 NEEDS_REPAIR 70-89 → 重跑诊断(最多3次)
    ❌ FAIL <70 → 报告用户

Step 6 · Reporter Sub-Agent

工程诊断报告生成
  • 加载所有产出: 上下文、本体、统计、验证、诊断、Judge反馈
  • 逐图分析: 11张图每个都标注 [OBSERVATION] 和诊断含义
  • 推理链综合: 从 R1-R8 重构推理轨迹
  • 强制章节: 执行摘要、统计验证、竞争假设披露、置信度
  • 输出: report.md (21章节) + run_summary.json
  • 每个结论必须 [OBSERVED] / [INFERRED] 标记

Step 7 · Report Reviewer Sub-Agent

物理真实验证 · 独立审计
  • 「你是有20年经验的高级工程师，对所有结论持怀疑态度」
  • 独立重算: 数据排序、去趋势、子组、Spearman vs Pearson
  • 6维度评分: 物理合理性、混杂控制、统计严谨性、逻辑、领域知识、可操作性
  • 结论:
    🏆 ENDORSED → 进入 Step 8
    🔄 CONDITIONAL/REJECTED → 4→5→6→7 修复循环(最多2次)
  • 输出: optimizer.md — 物理审计报告

Step 8 · Present Results Main Agent

结果展示 · 交付
  • 执行 artifact-check.mjs 验证所有生成文件完整性
  • 向用户展示:
    · 执行摘要 (2-3段)
    · 关键发现
    · 诊断类型 + 置信度
    · 建议行动 (P0/P1/P2)
  • 高亮报告Reviewer的 CONDITIONAL/REJECTED 问题
  • 交付工作区路径 + 所有产出文件清单
🤖 Agent深度解析 · 每一步对应哪个Agent、怎么做

🧠 Main Agent (Orchestrator)

负责任务: Step 0 · Step 1 · Step 2.5 · Step 8
作用: 流水线的主控编排器，负责全局协调、用户交互、质量管理
  • Step 0 — 创建目录、启动Python环境、拷贝输入数据
  • Step 1 — 执行 inspect.mjs、向用户提问、写入 user_context.json
  • Step 2.5 — 检查 clarification_needed.json、向用户追问未知参数含义
  • Step 8 — 执行 artifact-check.mjs、展示最终结果给用户
  • 质量管理 — 控制修复循环次数(全局≤5次)、判断何时需要停止
  • 文件解耦 — Agent之间仅通过工作区文件通信，不入主Agent上下文
  • 加载策略 — 渐进加载：执行到哪一步才读取哪一步的Agent指令

📚 Context Builder Agent

负责任务: Step 2 · 领域知识 + 本体构建
角色: 领域知识工程师。读取所有参考文档，构建工艺过程的完整本体，识别知识缺口
  • Step 0 - 加载用户上下文: 读取 user_context.json，提取工艺类型、已知问题、质量目标
  • Step 1 - 搜索参考目录: 递归读取SOP/手册/报告，提取: 设备参数、工艺阶段、故障模式、控制逻辑
  • 输出 → extracted_knowledge.json: 6大区块(设备/阶段/变量/故障/因果/产品等级)
  • Step 2 - 可选Web搜索: 最多5次查询，补足领域知识缺口，标注 [EXTERNAL KNOWLEDGE]
  • Step 3 - 构建本体:
    · 读取原始数据列名和类型
    · 为每个参数确定: 物理含义、控制类型(PID/手动)、测量位置、正常范围
    · 按参数重要性分类: CRITICAL / HIGH / MEDIUM / LOW
  • Step 4 - 识别知识缺口: 对未知物理含义的参数标记 UNKNOWN，生成 clarification_needed.json
  • 输出: ontology.json, schema.json, extracted_knowledge.json, clarification_needed.json

📊 Data Processor Agent

负责任务: Step 3 · 处理+统计+可视化+验证
角色: 诊断数据分析师。根据场景类型生成针对性的分析和可视化，使诊断师能从数据追踪因果链
  • Step 1 - 场景分类: 执行 inspect.mjs 了解数据，按检测标准分类为 CNC/连续/批次/热交换/质检/通用
  • Step 2 - 转换数据: convert.mjs → JSON格式
  • Step 3 - 预处理: 缺失值处理、异常值标记(IQR法)、排序验证、场景特定衍生特征
  • Step 4 - 统计分析:
    · stats.mjs: Pearson/Spearman/Granger/互信息/交互效应/分层分析
    · stats_validate.mjs: Simpson悖论/趋势混杂/异常值/变化点/分布分析
  • Step 5 - 异常检测: 自动识别异常区间、阈值偏离、过渡事件
  • Step 6 - 因果图: 构建验证过的因果边缘、共线性分组、根因候选
  • Step 7 - 自适应可视化: 根据场景类型生成不同的图表集:
    · CNC: 振动/Ra时间序列、进给力、磨损曲线
    · 连续: 温度剖面、张力梯度
    · 批次: 反应曲线、批间变异
    · 热交换: HTC衰减、污垢电阻
  • 关键输出: feature_summary.json, validate_report.json, causal_evidence_map.json, plot_manifest.json

🔍 Diagnostician Agent

负责任务: Step 4 · 核心诊断推理
角色: 根因分析师。通过物理因果链追踪诊断工业异常。数据是证据，物理是法官。
  • Phase 0 - 加载所有证据: 读取9类产出文件，先读验证报告再形成假设
  • Phase 1 - 直接数据探测:
    · Probe 1-3: 工具段检查、段分析、梯度分析
    · Probe 4: 阈值验证 (当粒度跨越阈值时质量是否跳变)
    · Probe 5-6: 物理定量计算 (热膨胀、挠度模型)
    · Probe 7-11: 可区分性分析、过渡分析、子组属性
  • Phase 2 - 产品分层分析: 验证每个产品等级内相关方向一致
  • Phase 3 - 5步推理协议:
    A. 生成假设 + 7步物理链 + 可视化证据
    B. 交叉验证: 数据探针 × 异常区间 × 因果图
    C. 可区分性矩阵: 每个假设对 DISCRIMINABLE / INDISTINGUISHABLE
    D. 排除: 物理不可能性 + 数据矛盾 + 综合排除
    E. 结论: DETERMINED / COMPETING_SET / NEEDS_DATA
  • 输出推理链R1-R8:
    R1: 领域知识 + 已知故障模式
    R2: 数据探针发现 (11项探针)
    R3: 验证过滤 (排序/Simpson/趋势/异常值/变化点)
    R4: 假设生成 (4+N竞争假设)
    R5: 可区分性评估
    R6: 排除文档
    R7: 诊断结论 + 物理迹
    R8: 不确定性分解
  • 规则: 每个假设必须可证伪；量级必须匹配；IDISTINGUISHABLE对置信度≤65

⚖️ Judge Agent

负责任务: Step 5 · 质量评审门
角色: 最终质量门。交叉验证诊断推理与统计验证报告，检查10个维度的质量。
  • Step 0 - 加载所有产出: ontology、feature_summary、validate_report、diagnosis、evidence、confidence、reasoning_chain
  • Step 0.5 - 交叉验证 (NEW — 强制):
    1. 排序验证: time_sorted=false时, 滞后相关必须声明排序伪影
    2. Simpson悖论: 忽略方向反转 → BLOCKING
    3. 趋势混杂: 衰减>50%必须报告去趋势r
    4. Spearman vs Pearson: 偏态分布必须声明
    5. 异常值驱动: 标记必须讨论
    6. 参数含义: 未解决CRITICAL → BLOCKING
    7. 可区分性: 未检查→BLOCKING, INDISTINGUISHABLE置信度>65→BLOCKING
  • 评分10维度 (每项0-10, 加权):
    · 数据质量意识 (10%) · 变量分类 (10%)
    · 时间对齐 (10%) · 可视化质量 (5%)
    · 基于证据 (20%) · 推理链质量 (15%)
    · 因果vs相关 (10%) · 不确定性披露 (10%)
    · 报告质量 (5%) · 不过度宣称 (5%)
  • 裁决: PASS ≥90 / NEEDS_REPAIR 70-89 (最多3次) / FAIL <70
  • 输出: judge_feedback.json

📝 Reporter Agent

负责任务: Step 6 · 报告生成
角色: 工程报告撰写人。将诊断推理转化为自包含、可审计的工程文档。
  • Step 0 - 加载所有产出: 读取14类文件(用户上下文/本体/数据质量/统计/可视化/诊断/Judge反馈)
  • Step 1 - 逐图理解 (强制):
    · 从 image_captions.json 读取结构化描述
    · 每张图提取: 展示内容、关键观测、趋势形状、异常区域、诊断含义
    · 验证图标注: 统计验证图 → CCF窗口/分层相关/去趋势/Spearman/异常值
  • Step 1.5 - 推理链综合:
    · 从 reasoning_chain.json 提取 R1-R8 每步的关键发现
    · 区分 [OBSERVED] 和 [INFERRED]
    · 生成幻觉审计日志(附录E)
  • Step 2 - 生成报告 (21个章节):
    · 执行摘要 · 推理概述(7个分段) · 分析目标 · 用户上下文
    · 工业本体 · 数据描述 · 变量分类 · 预处理
    · 逐图分析(11张图) · 诊断结果 · 根因综合
    · 统计验证(强制) · 竞争假设披露(强制) · 置信度 · 不确定性

  • 规则: 每个结论引用证据来源+等级；每张图必须嵌入；统计验证在前台展示不埋没在附录
  • 输出: report.md, run_summary.json

🔬 Report Reviewer Agent

负责任务: Step 7 · 物理真实验证
角色: 「20年经验的高级工程师」— 最重要的质量门。不做内部一致性检查(Judge的职责)，而是做物理现实校验。
  • Step 0.5 - 加载资源 + 读取原始数据(不信任流水线摘要，直接检查原始CSV)
  • Step 1 - 物理机制验证:
    · 物理合理性: 机制能否产生观测症状？
    · 量级匹配: 原因的量级能产生效果的量级吗？(eg: 1-2°C在75°C能否产生可检测的降解？用Arrhenius验证)
    · 时间尺度匹配: 退化时间线是否与已知物理一致？
    · 症状完整性: 机制是否解释所有观测？
  • Step 1.1 - 领域特定量化验证:
    · CNC: Ra≈f²/(32×rε)、ISO 10816振动等级、Taylor刀具寿命方程
    · 薄膜: Arrhenius热降解、PET环状三聚体形成、拉伸物理
    · 热交换: 污垢电阻Rf、HTC衰退一致性
  • Step 1.2 - 推理链幻觉检测:
    · 8种模式: 模糊量化/无锚推断/缺少替代/不可证伪/证据等级膨胀/置信度过高/忽略矛盾/制度盲点
    · 随机抽取3个结论追溯检查
  • Step 2 - 独立统计验证:
    · 独立去趋势、子组分析、排序验证、滞后分析
    · 5种混杂模式: 产品等级/时间趋势/批次排序/遗漏变量/反向因果
  • Step 3 - 统计谬误审计: 趋势相关? 子组稳定? 非线性? 异常值敏感? 多重检验?
  • 6维度评分: 物理合理性/混杂控制/统计严谨性/逻辑连贯/领域知识/可操作性
  • 裁决: ENDORSED(所有≥7) / CONDITIONAL(1-2个<7) / REJECTED(≥3个<7)
  • 输出: optimizer.md
🔄 修复循环机制

修复循环详解

循环触发条件最大次数重跑步骤全局上限
Judge 修复 Judge评分: NEEDS_REPAIR (70-89) 3次 Step 4 → 5 (诊断→评审) 全局总重诊断 ≤ 5次
Reviewer 修复 Reviewer裁决: CONDITIONAL/REJECTED 2次 Step 4 → 5 → 6 → 7 (完整流水线)
→ Judge PASS ≥90 → 正常进入 Step 6，不触发修复
→ Judge NEEDS_REPAIR 70-89 → re-spawn Diagnostician，附带 REPAIR_INSTRUCTIONS 中的修正要求
→ Judge FAIL <70 → 直接报告用户，不继续流水线
→ Reviewer CONDITIONAL/REJECTED → 从 Step 4 开始重新跑(含修正)，再经过 5→6→7
→ diag_iters ≥ 5 → 所有修复循环强制停止
⚠️ 修复循环次数 持久化存储在 .pipeline_events.jsonl 中，不依赖内存状态
📊 证据等级体系 (v4.2)
等级来源置信度标签规则
1数据中的直接测量值最高[Evidence Rank 1]最可靠，来自数据本身
2用户提供的文档 (SOP/手册)高[Evidence Rank 2]需要确认文档有效性
3统计分析 + 验证报告中-高[Evidence Rank 3]通过Simpson/趋势/异常值验证才有权重
4图表可视化证据中[Evidence Rank 4]目视确认的趋势/发散/异常区域
5领域知识 / 工艺逻辑中[Evidence Rank 5]需要有定量物理支撑
6外部Web参考资料低[Evidence Rank 6] [EXTERNAL]必须明确标注 EXTERNAL
7未验证的假设最低[Evidence Rank 7]不能作为主要结论依据

结论上限 = 最弱证据等级。例如 统计发现(r=0.99 Rank 3) + 物理机制(Rank 5) → 受Rank 5限制。
统计证据(等级3)现在包含验证状态: 通过所有验证的相关>未通过的相关。

🚫 反推测规则 (Anti-Speculation Rules)

禁止行为 (NEVER)

  • 声称"X导致Y"而未满足全部5大因果关系标准(时间顺序+统计显著+滞后一致+物理机制+无子组矛盾)
  • 假设未测量变量的行为
  • 混淆相关与因果
  • 将假设呈现为既定事实
  • 置信度超出证据支持的范围
  • 使用滞后相关作为因果证据而数据未按时间排序
  • 声称聚合相关在主导子组中方向反转时仍有意义
  • 引用原始相关而不检查存在时间趋势时的去趋势相关
