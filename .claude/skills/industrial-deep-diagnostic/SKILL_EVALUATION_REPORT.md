## 二、架构设计分析

### 2.1 整体架构

该 Skill 采用**分层流水线 + 多智能体协作**架构，核心设计理念为「证据优先、物理验证、假设竞争」。

```
┌─────────────────────────────────────────────────────┐
│                   主代理 (Orchestrator)                │
│  管理 Step 0-8 全流程，调度子代理，控制修复循环       │
└────────┬────────┬────────┬────────┬────────┬────────┘
         │        │        │        │        │
    ┌────▼───┐┌──▼────┐┌──▼────┐┌──▼────┐┌──▼────┐
    │Context ││Data   ││Diag-  ││Judge  ││Report │
    │Builder ││Proc.  ││nost.  ││       ││Review.│
    └────────┘└───────┘└───────┘└───────┘└───────┘
```

**优点：**
- 关注点分离清晰：每个代理有明确定义的职责边界
- 数据流可追溯：通过 JSON 工件在代理间传递，每个阶段有明确的输入/输出契约
- 并行化潜力：Context Builder 和 Data Processor 可部分并行执行

**不足：**
- 主代理编排逻辑未以可执行脚本形式固化，依赖 LLM 理解 SKILL.md 后手动执行步骤，存在执行偏差风险

### 2.2 目录结构与工件管理

产出目录按功能分层（`00_input/` → `01_ontology/` → `02_processed/` → `03_figures/` → `04_diagnostics/` → `05_review/` → `06_scripts/`），每个目录有明确的 JSON Schema 约束。这种设计类似于 DAG 工作流引擎的中间表示（IR），使得任何阶段的产物都可以被独立检查和复现。

**优点：**
- 工件完整性可通过 `artifact-check.mjs` 自动校验
- 时间戳命名的 run 目录支持多轮诊断对比
- JSON Schema 验证（9个）保证了结构化输出的质量

## 三、执行流程分析

### 3.1 流水线步骤（Step 0-8）

| Step | 名称 | 代理 | 核心产出 |
|------|------|------|----------|
| 0 | Setup | 主代理 | run 目录、输入清单 |
| 1 | Inspect | 主代理 | 数据探查报告 |
| 2 | Context | Context Builder | 本体、Schema、知识缺口 |
| 3 | Data+Viz | Data Processor | 统计特征、验证报告、可视化、异常标注 |
| 4 | Diagnose | Diagnostician | 诊断、证据链、置信度、推理链 |
| 5 | Judge | Judge | 评审反馈（10维度评分） |
| 6 | Report | Reporter | Markdown 诊断报告 |
| 7 | Audit | Report Reviewer | 物理真实性审计、优化建议 |
| 8 | Present | 主代理 | 最终呈现 |

**流程设计亮点：**

1. **澄清门控（Clarification Gate）**：Step 2 完成后，Context Builder 生成 `clarification_needed.json`。若存在 CRITICAL 级别的未知参数物理含义，流程必须暂停等待用户澄清——确保后续诊断不会基于错误假设。这是工业诊断中极其重要的安全机制。

2. **修复循环（Repair Loop）**：Judge 评分 < 90 时触发修复，最大 3 次，且全局迭代上限 5 次。Report Reviewer 则允许 2 次优化迭代。这种双重修复循环设计防止了无限自我修正。

3. **统计前置验证（Stats First）**：在诊断代理启动前，Data Processor 必须完成统计验证（Simpson's Paradox、趋势混淆、离群值驱动），确保诊断基于已验证的证据。

### 3.2 流程评估

| 方面 | 评价 |
|------|------|
| 步骤粒度 | 适中。8 步对于工业诊断场景是合理的，既不过细（避免碎片化），也不过于粗粒度（保留可追溯性） |
| 依赖关系 | 大部分步骤严格串行（Step 2→3→4→5→6），但 Step 1 和 Step 2 的参考搜索可并行 |
| 错误恢复 | 修复循环设计合理，但缺少对中间步骤（如数据预处理失败）的自动重试机制 |
| 流程可视化 | SKILL.md 中的 ASCII 流程图清晰，但缺少 Mermaid/Graphviz 格式的自动化流程图 |

### 4.2 Diagnostician — 核心推理引擎（评分 9.5/10）

Diagnostician 是整个 Skill 的核心创新所在。其设计超越了传统的「跑统计→看相关性→给结论」模式，引入了以下独特机制：

**Phase 1：直接数据探测（Data Probing）**
这是该代理最关键的差异化能力。它不是读取统计汇总表，而是直接探查原始数据来验证物理假设。包括：
- 异常区间参数行为检查
- 过渡事件分析（工具更换后质量是否重置？）
- 物理阈值验证（临界点效应）
- 共线性打破（通过分组寻找区分信号）

**Phase 2-3：产品分层 + 候选参数筛选**
通过 `UNIVERSAL` / `CONSISTENT_SIGN` / `BETWEEN_PRODUCT_ONLY` / `SIMPSON_REVERSAL` 四类分类，系统性地排除伪相关性。

**Phase 4：5步竞争假设协议**
```
Step A: 物理逻辑链生成（ROOT CAUSE → PHYSICAL VARIABLE → PROCESS STATE → INTERMEDIATE → DEFECT）
Step B: 竞争假设评分（统计+物理+时间+无混淆+症状完整性，5因子加权）
Step C: 可区分性审查（v6.0新增）—— 核心创新
Step D: 因果合成
Step E: 最终置信度评估
```

**Step C（可区分性审查）的创新价值**：
- 明确检查竞争假设是否预测不同的可观测模式
- 对不可区分的假设对，输出 `COMPETING_SET` 而非伪选一个
- 指定何种区分数据可以解决歧义
- **置信度上限 65 规则**：对不可区分的竞争假设，单假设置信度不能超过 65

这一设计有效解决了 LLM 诊断中的核心问题——「伪确定性幻觉」：当数据无法区分两个假设时，模型倾向于随意选择一个。

**定量可行性检查**：

| 检查类型 | 适用场景 | 公式 |
|----------|----------|------|
| 热膨胀 | 温度→尺寸误差 | ΔL = α × L × ΔT |
| Arrhenius 动力学 | 温度→化学降解 | k(T₁)/k(T₂) = exp(Ea/R × (1/T₂ - 1/T₁)) |
| 能量平衡 | 功率→温升 | ΔT = P × t / (m × Cp) |
| Taylor 刀具寿命 | 刀具磨损 | VT^n = C |

这些检查将诊断从「定性推理」提升到「定量验证」，是其区分于普通诊断技能的关键。

**推理链（R1-R8）结构化追踪**：
```
R1: Data Characterization → R2: Statistical Discovery → R3: Validation Filter
→ R4: Hypothesis Evolution → R5: Causal-Symptom Amplification
→ R6: Alternative Elimination → R7: Uncertainty Decomposition → R8: Synthesis & Conclusion
```

8段推理链的输出在 `reasoning_chain.json` 中完整记录，每段包含 `inputs`、`reasoning`、`outputs`、`alternatives_considered`、`uncertainty`、`falsification_condition`。这相当于给了后续的 Judge 和 Report Reviewer 一个完整的「思维审计追踪」。

### 4.3 Judge — 方法论质量守门人（评分 9.5/10）

Judge 的 10 维度评分体系是其核心力量：

| 维度 | 权重 | 评估内容 |
|------|:----:|----------|
| 数据质量意识 | 10% | 缺失值、离群值、排序验证 |
| 变量分类 | 10% | 与本体一致、分组列识别 |
| 时间对齐与排序 | 10% | 时间排序验证 |
| 可视化质量 | 5% | 标签、图例、统计验证图 |
| **证据基础结论** | **20%** | 每项结论是否引用证据源 |
| **推理链质量** | **15%** | 完整性、证据接地、反事实、不确定性 |
| 相关 vs 因果 | 10% | Simpson's Paradox 排除、趋势混淆检查 |
| 不确定性披露 | 10% | 置信度、证据缺口、排序/分层/趋势说明 |
| 报告质量 | 5% | 模板使用、自包含性 |
| 完整性 | 5% | 所有必需产出存在 |

**Step 0.5（交叉引用验证报告与诊断）** 是 Judge 的关键创新——它不只是看诊断报告本身，而是系统性地交叉对比验证报告中的发现，检查诊断是否：
- 承认排序问题
- 处理 Simpson's Paradox
- 使用降趋势相关性
- 标记离群值驱动的相关性

这种「检测一致性」而非仅「检测格式」的方法论大大提高了虚假过关的难度。

**评分上限规则**：
- 数据未按时间排序且使用滞后相关作为主要证据 → 分数上限 85
- 不可区分竞争假设赋予 >65 置信度 → 分数上限 65

### 4.4 Report Reviewer — 物理真实性审计师（评分 9.0/10）

这是流水线中最独特的角色。它不是内部一致性检查（那是 Judge 的工作），而是「外部物理现实检查」。

**核心审计维度**：
1. **物理机制验证**：构建从第一性原理的机制链，检查物理可行性
2. **量级匹配**：效应量级是否与原因量级匹配？（例如，1-2°C 温差在 75-80°C 能否产生可检测的热降解？）
3. **时间尺度匹配**：降解时间线是否与已知物理一致？
4. **领域特定定量验证**：针对 BOPET/BOPP 薄膜、CNC 加工、换热器等场景的特定物理检查
5. **推理链审计**：检测模糊量化、无锚推断、缺失替代假设、不可证伪结论、证据等级膨胀、置信度过高、忽略矛盾、区间盲视等 8 种幻觉模式
6. **抽查协议**：随机选 3 个结论，逐层追溯其证据基础

这一角色使得诊断不仅「内部自洽」且「外部真实」——它是防止「看起来很对但实际错误」诊断的最后防线。

### 4.5 Data Processor — 数据分析专家（评分 8.5/10）

**亮点**：
- **场景驱动分析**：不是通用图表生成器，而是基于场景分类（CNC、连续过程、间歇化学、热交换、质检、通用）自适应选择分析方法
- **强制异常检测**：Python 脚本实现自适应阈值、突变检测、异常区间合并
- **过渡事件分析**：检测分类变量变化前后的质量跳变——这是区分组件级退化和系统级退化的关键

### 4.6 Context Builder — 领域知识工程师（评分 8.5/10）

**亮点**：
- **三步知识获取**：参考文档搜索 → 网络调研 → 用户澄清
- **参数物理含义分类**：KNOWN / INFERRED / UNKNOWN 三级分类
- **强制澄清门控**：CRITICAL 级别的未知参数必须暂停流程
- **变量角色标注**：target / predictor / confounder / control / metadata

## 五、统计验证框架分析（评分 9.0/10）

### 5.1 验证体系

该 Skill 的统计验证框架是其区别于通用数据分析技能的核心竞争力。它不仅仅跑统计，而是系统性地检测统计陷阱：

| 验证方法 | 检测目标 | 工具 |
|----------|----------|------|
| 排序验证 | 数据是否按时间排序 | `stats.mjs` |
| Simpson's Paradox 检测 | 聚合相关在子组中是否反转 | `stats_validate.mjs` |
| 趋势混淆检测 | 相关性是否由共同时间趋势驱动 | `stats_validate.mjs` |
| 离群值敏感性 | 相关性是否被极端值驱动 | `stats_validate.mjs` |
| Spearman vs Pearson 对比 | 相关性对方法选择是否鲁棒 | `stats.mjs` |
| 多重检验校正 | 多少「显著」结果是随机期望的 | Bonferroni |
| 滞后窗口一致性 | CCF 是否在相邻滞后持续存在 | `stats.mjs` |
| 变化点检测 | 是否存在区间转换 | `stats_validate.mjs` |
| 交互效应检测 | 参数间是否存在协同效应 | `stats.mjs` |
| Granger 因果检验 | 时序预测能力 | `stats.mjs` |
| 互信息 | 非线性依赖 | `stats.mjs` |

### 5.2 置信度调整协议

```
数据未时间排序 + 使用滞后 → -25 到 -40
Simpson's Paradox（主导子组中反转）→ -20 到 -30
趋势混淆（衰减 > 50%）→ -15 到 -20
离群值驱动 → -10 到 -15
Spearman-Pearson 分歧 > 0.15 → -5 到 -10
```

这种量化调整规则将置信度从主观估计变为可计算、可审计的指标。

### 5.3 证据等级体系（7级）

1. 直接测量（最高）
2. 用户提供的文档
3. 统计分析（含验证状态）
4. 图表视觉证据
5. 既有过程逻辑/领域知识
6. 外部网络引用
7. 假设（无支持，最低）

每个结论受其最弱证据等级限制——这确保了即使有强统计相关性，如果物理机制不明，也不会被高估。

## 六、反投机规则评估（评分 9.0/10）

### 6.1 禁止行为（9条）

- 未满足全部 5 条因果标准时声称「X 导致 Y」
- 假设未测量变量的行为
- 混淆相关与因果
- 将假设呈现为既定事实
- 超越证据支持地过度声称置信度
- 数据未时间排序时使用滞后相关作为因果证据
- 声称聚合相关有意义但在主导子组中反转
- 在存在时间趋势时不检查降趋势相关
- **忽略变化点/区间转换证据**

### 6.2 强制行为（7条）

- 每个结论披露置信度
- 指出证据缺失或不足
- 分离数据推导结论和领域知识推断
- 证据不足时建议进一步验证
- 根因不确定时呈现多个假设
- 报告验证发现与统计证据并排
- 关键相关性上承认排序/分层/趋势说明

### 6.3 语言模板（5种标记）

[OBSERVATION] / [INFERENCE] / [HYPOTHESIS] / [UNCERTAINTY] / [VALIDATION]

这些模板强制区分「观察到的事实」和「推断出的结论」，有效防止虚幻的确定性。

## 八、主要优势总结

1. **物理因果推理 vs 统计相关性**：该 Skill 的核心区分点是它不满足于「参数 X 与缺陷 Y 高度相关」，而是强制要求物理机制链、定量可行性检查、数据探测验证。这是工业诊断和学术数据分析的本质区别。

2. **假设竞争协议 + 可区分性审查（v6.0）**：明确检查竞争假设是否预测不同可观测模式，对不可区分假设输出 `COMPETING_SET`，这是诊断方法论的重大进步，直接针对 LLM 的「伪确定性」幻觉问题。

3. **双重质量控制**：Judge（内部一致性） + Report Reviewer（外部物理真实性），两者互补，构成完整的质量保障体系。

4. **场景驱动的自适应分析**：不是一刀切的统计流程，而是根据 CNC、连续过程、间歇化学等场景自适应调整分析策略。

5. **过渡事件分析**：通过检测工具更换/材料切换前后的质量跳变来区分组件级和系统级退化——这是一个领域专家级别的诊断技术。

6. **推理链完整审计追踪**：R1-R8 的 8 段结构化推理记录 + `reasoning_chain.json`，使得诊断过程完全可追溯和可审计。

## 十、改进建议（按优先级排序）

### 高优先级

1. **实现确定性编排引擎**：将 Step 0-8 的流转逻辑实现为可执行脚本（如 `pipeline.mjs`），而非依赖 LLM 解读文本。这消除了执行偏差风险，也使 Skill 可在无人工干预的批处理模式下运行。

3. **添加故障排查指南**：常见失败模式（如数据文件不存在、Schema 验证失败、Judge 持续给低分等）的处理文档。

### 中优先级

5. **添加预测性维护模块**：基于诊断结果生成退化趋势预测和预防性维护建议。

6. **添加贝叶斯更新机制**：在多次诊断运行中累积对假设的信念更新，支持持续改进。

8. **增加领域知识贡献模板**：提供标准化的领域物理验证模板，支持社区贡献新的过程类型。

### 低优先级

## 十二、结论

**Industrial Deep Diagnostic Skill 是当前工业诊断领域最先进的 LLM-based 诊断系统之一。** 它在以下几个方面建立了行业基准：

1. **物理因果推理的深度和严谨性** — 通过假设竞争协议、定量可行性检查和可区分性审查，将 LLM 诊断从「统计模式匹配」提升到「物理因果演绎」。

2. **质量控制的层次化设计** — Judge + Report Reviewer 的双重审查，从内部一致性和外部物理真实性两个维度保障诊断质量，这在同类系统中是独一无二的。

**综合评价**：该 Skill 体现了从「通用数据分析」到「工业级物理诊断」的范式升级。其设计理念——特别是假设竞争协议和可区分性审查——值得推广到更广泛的 LLM 推理场景中。
